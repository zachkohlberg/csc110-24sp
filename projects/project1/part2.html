Guided Project 1.2

This program will prompt for additional information, perform calculations closer to the real D&D rules, check whether user input is valid, and print a nicer-looking character sheet.

Learning Objectives

  • Use conditional statements to check for invalid input
  • Use the printf function to format output
  • Use conditional statements to change how a program calculates variables and formats its output
  • Modify an existing program to meet new requirements

Prerequisites and Setup

  • You should have already completed GP 1.1, including debugging the program until it passes all of the tests.
  • Create a file named CharacterSheet2.jsh in your project1/part2 folder. Copy all of the code from project1/part1/CharacterSheet1.jsh into this new file.

Tasks

Task 1: Guided Coding

Follow the steps below and write the program as directed. Test your code frequently, at the bare minimum performing every test given in the steps.

If your code is not working as expected, do not move on to the next step until you have fixed it. Moving on without finishing a step correctly will make the problem significantly harder to fix and likely add new problems as you continue working. If you disregard this advice, then you may get to the end of the steps and find it would actually be less work to start over from step 1 than to debug the code you just finished writing.

  1. This program requires two additional pieces of input. Prompt the player for their character's ancestry and class after the character name and before the level. Test these new prompts like you tested the prompts in 1.1: run the program, make sure the prompts appear, and make sure the variables have the expected value after all of the prompts are finished.

  2. If the user enters invalid input, then our program must print an error message and exit. You should modify your prompts to tell the user what input is allowed for the ancestry, class, level, and ability scores. This will be a helpful reminder when you test the program.

    Note: You can check each input value immediately after the prompt, or you can check all of them in a separate section of your program after all of the input is finished. The first option will give a better user experience because you will know right away when you make a mistake, and you won't waste time entering more input before the program exits. It also makes your program faster to test for the same reasons. The second option makes it a little easier to organize the code, and all of the error-checking could be combined into one if statement. I recommend the first option because I believe you'll benefit the most from the ease of testing. My directions will only describe how to code this option.

  3. Add an if statement after the level prompt. A valid level is an integer between 1 and 20, inclusive. We want to check whether it is invalid, so we need a condition that asks whether the level is outside of this range. An invalid level is either less than 1 or greater than 20. There are two parts to this condition, so we need to join them with an AND (&&) or an OR (||). In this case, we should use OR. There are a couple of ways to determine that OR is appropriate: the wording of the condition when we describe it in English only makes sense with the word "or", and we only need to meet one of these two conditions for the level to be invalid (it's not even possible for it to be both less than 1 and greater than 20 at the same time). In the body of our if statement, we need to print an error message and exit our program. The exact error message is ERROR: INVALID INPUT, and we can exit the program with System.exit(1);. Any integer is a valid argument for the System.exit function, but a value other than 0 means the program encountered an error. All together, the prompt with error checking would look something like this:

// prompt for input
System.out.println("level prompt message");
int variableName = nextInt();

// check for invalid input
if (/*TODO: condition*/) {
    println("ERROR: INVALID INPUT");
    // The /exit command won't work here because this whole block is part of a
    // Java code snippet. Jshell expects everything in a snippet to be valid
    // Java code.
    exit(1);
}

  1. You can use a similar prompt to check each ability score. The ability scores must be integers between 8 and 15 inclusive. After you add the error-checking for the level and abilities, you should test each of them a few times. Try a few values inside and outside the range, and make sure the program exits with the error message if and only if you enter a value outside the range. In particular, you should check the boundaries of the range. One of the most common mistakes in this type of code is to make a range inclusive when it should have been exclusive, or to make it exclusive when it should have been inclusive. If you make that mistake, then your code will work correctly for all but one or two numbers. You should check 0, 1, 20, and 21 for the level, as well as 7, 8, 15, and 16 for each ability score.

  2. Once your error-checking code works for the numbers, you also need to check the ancestry and class. Each of these have three options, and the user's input is invalid if it is not equal to one of these options. This means we have three parts to our condition, and they must be joined with AND or OR. Which is appropriate? Let's try describing invalid input more closely to how we would code it (checking each possibility separately) and see whether AND or OR is more natural: an invalid ancestry is not an elf and it is not a dwarf and it is not a human. Therefore we should use AND (&&).

    However, this can be tricky. There's another way to phrase our condition that sounds natural and uses OR instead of AND: "an invalid ancestry is not an elf, dwarf, or human". Does this mean it doesn't matter whether we use AND or OR? No, it still matters. The trick with this phrase is how it looks when structured like code: "it is not true that (an invalid ancestry is an elf or it is a dwarf or it is a human)." We are using or to join several equality checks (instead of inequality checks), then applying the NOT (!) to the entire expression, which I've indicated by wrapping most of the expression in parenthesis.

    You'll develop an intuition for boolean logic with practice, so don't worry if you had a hard time with this part. I'll summarize everything you need to know for the AND solution to this error check:

    • Check whether the ancestry/class is NOT equal to each option.
    • Join each of these comparisons with AND (&&).
    • Never compare strings with s1 == s2 or s1 != s2. This will not work as you want it to. Instead, compare them using the .equals function: s1.equals(s2) for equality and !s1.equals(s2) for inequality.
    • Valid ancestries are "elf", "human", and "dwarf".
    • Valid classes are "fighter", "rogue", and "ranger".
    • Use the lowercase versions of these names. You're free to assume all the user's input for class and ancestry will be lowercase, and I will only use lowercase input for these traits when testing your code. However, if you want your program to ignore case, then you can modify the code that reads their input for class and ancestry by changing nextLine() to nextLine().toLowerCase(). This will convert any capital letters in the user input to lowercase letters. Another solution is to use .equalsIgnoreCase instead of .equals, which will let you check for equality while preserving the user's capitalization. If we use .equalsIgnoreCase and don't convert the input to lowercase, then our output will use the same capitalization as the user. This may or may not be desirable; it depends on how you want the output to look.

    Remember to test your new error-checking code. You should now get an error message if you don't enter a valid ancestry or class, but the program should still work if you enter valid input for everything.

  3. Now we need to update the calculations. We can break this into three phases: adjust ability scores, calculate ability modifiers, and calculate other values. We'll start with adjusting the ability scores. Each ancestry changes one or more ability scores. Humans increase all of their scores by 1, elves increase their dexterity by 2, and dwarves increase their constitution by 2. You can structure this part as an else-if chain:

if (variableName.equals("ancestry1")) {
    // TODO: adjust ability scores for ancestry 1
} else if (variableName.equals("ancestry2")) {
    // TODO: adjust ability scores for ancestry 2
} else if (variableName.equals("ancestry3")) {
    // TODO: adjust ability scores for ancestry 3
} else {
    // We don't need this, but it can make debugging easier. If we made a
    // mistake in our error-checking code and allowed an invalid ancestry, or
    // if we add more ancestries later and forget to update this part, then
    // we'll see this error message and know we made a mistake. This error
    // message is for us, not the user. We should only see it if our code is
    // wrong.
    println("ERROR: UNRECOGNIZED ANCESTRY! SKIPPING ADJUSTMENTS");
}

  1. After you adjust ability scores for the character's ancestry, you need to calculate each ability's modifier. This is a separate number from the score, and it's used in most calculations that care about the ability instead of the raw score. The Project 1 Background section explains the difference between ability scores and modifiers in more detail.

    The SRD has a table showing the modifier for each ability score. However, we can more easily calculate this with a short formula: abilityModifier = (abilityScore / 2) - 5. The parenthesis aren't necessary, but they emphasize the order of operations. Dividing after subtraction ((abilityScore - 10) / 2) would not give us the correct modifier for some scores below ten.

  2. The Calculations section covers the remaining calculations you need to perform. You can find the relevant formulas and other information there. Some of these calculations change depending on the class, so you should use an else-if chain similar to what you wrote for the ancestry adjustments. However, make sure you declare any variables before the start of the if-else chain. If you declare a variable inside an if or else body, then that variable will only exist inside that code block, and it won't be available later when you perform other calculations or print your output. Here's an outline to get started:

// Declare all new variables here
int hp = 0;
// TODO: other variables

if (variableName.equals("class1")) {
    hp = /*TODO: class 1 hp calculations*/;
    // TODO: other class 1 calculations
} else if (variableName.equals("class2")) {
    // TODO: class 2 calculations
} else if (variableName.equals("class3")) {
    // TODO: class 3 calculations
} else {
    // As with the ancestry section, the extra else can help with debugging
    println("ERROR: UNRECOGNIZED CLASS! SKIPPING CALCULATIONS");
}
  1. Test your code. We haven't updated the output yet, but you can run the program and check the value of each variable from the calculations section to make sure it's what you were expecting.
  2. Finally, you can adjust the output to follow the new format for 1.2 and show all of the new information you've calculated. Two of the formatting requirements can be tricky to figure out: adding a plus to any nonnegative modifiers and right-aligning the ability scores. You can use if statements to handle this, but the printf function solves this problem much more easily than an if statement. I won't explain printf here, so read the notes if you aren't familiar with it. The formatting specifiers you'll need are %2d to right-align an ability score and %+d to force a plus or minus sign in front of an ability modifier.

Additional Details

I/O Format

A Guided Project program must meet the following I/O requirements:

  • The program accepts the required input in the order shown in the assignment and does not expect any additional input.
  • The program produces each line of required output in the order specified in the assignment.

The program may include additional output before, between, and after lines of required output. This additional output must be on a separate line from any required output.

Your program must accept the following user input:

  • The player's name (String)
  • The character's name (String)
  • The character's ancestry (String)
  • The character's class (String)
  • The character's level (int)
  • The character's ability scores
    • Strength (int)
    • Dexterity (int)
    • Constitution (int)

If all user input is valid, then your program must print the following output:

  • Player: <NAME>
  • Character: <NAME>, level <LEVEL> <ANCESTRY> <CLASS>
  • HP: <HP>, AC: <AC>
  • Ability Scores:
  • Str <STRENGTH SCORE> <STRENGTH MODIFIER>
  • Dex <DEXTERITY SCORE> <DEXTERITY MODIFIER>
  • Con <CONSTITUTION SCORE> <CONSTITUTION MODIFIER>
  • Attacks:
  • <ATTACK1 NAME> <ATTACK1 MODIFIER>
  • <ATTACK2 NAME> <ATTACK2 MODIFIER> (note: some characters only have one attack)

Two additional requirements:

  • If a score (strength, dexterity, and constitution) is a single-digit number, then it must be padded with an extra space so that all scores are right-aligned in their column.
  • Each modifier (strength, dexterity, constitution, and attack) must be preceded by a plus or minus sign. Negative modifiers have a minus sign, and nonnegative (which includes zero) modifiers have a plus sign. This reinforces how these numbers are meant to be used: they are usually added to dice rolls and other values rather than used on their own.

The ability scores and attacks should also be indented (as shown in the example below), but this will not be checked by the tests.

Here is an example of possible user input and the resulting character sheet:

Output:

Valid Input

There are two ways for the user to enter invalid input:

  1. The user can enter a string or double when you were expecting an integer. If the user entered the word "nine" or the number 12.4 when asked for their level, then this would be the wrong type of input.
  2. The user can enter the correct type of input, but not one of the values they were allowed to enter. If the user entered -2 or 50 when asked for their level, then this would be the correct type of input (an integer), but not one of the integers they were allowed to choose for their level (a number from 1 to 20).

The first kind of error is more complicated to deal with, and you probably won't learn how to handle those errors until later in the semester. You only need to worry about the second kind of error in this assignment. You can assume that the user will always enter the correct type of value, but your code must check whether it was one of the values they were allowed to choose. If the user enters any invalid input, then your program must print the message ERROR: INVALID INPUT and exit.

Here are all of the rules you must enforce when checking the user's input:

  • The level must be an integer from 1 to 20.
  • Each ability score must be an integer from 8 to 15.
  • The ancestry must be either "human", "elf", or "dwarf".
  • The class must be either "fighter", "ranger", or "rogue".

You can assume that the user's input for class and ancestry are always lowercase. Your code must correctly handle the lowercase versions of these strings, but it doesn't matter what you do when given other capitalizations (such as "Human", "RANGER", or "DwARf"). You can treat those as invalid and just print an error message (easiest to code but less user-friendly), or you can make your code case-insensitive so that the capitalization does not matter (harder to code but more user-friendly). However you decide to handle the input, the ancestry and class must appear as all lowercase in the program's output.

Calculations

This program involves more calculations and variables than the previous program. This section will serve as a reference for all values that are calculated by the program.

  1. Before any other calculations, some of the ability scores must be adjusted by the player's ancestry:
    • If the character is a human, increase each ability score by one.
    • If the character is an elf, increase their dexterity score by two.
    • If the character is a dwarf, increase their constitution score by two.
  2. Each ability score has an associated number called its modifier. The modifier does not alter the score, it is derived from the score. The srd link has a table showing the modifier associated with each score from 1 to 30, but we can calculate the modifier with a formula.
    • Each ability modifier is calculated by dividing the score by 2 and subtracting 5. The formula should look like this: modifier = score / 2 - 5.
    • It's important that you divide before subtracting because integer division will round towards zero, not down. We could perform a similar calculation by subtracting 10 and then dividing by 2, but this would give the wrong value for odd ability scores below 10. For example, a score of 9 would produce a modifier of 0 when it should have a modifier of -1.
  3. The character's HP equals their level multiplied by the sum of their constitution modifier and their "hit die". The hit die is determined by the character's class. A fighter has a hit die of 10, and the other two classes have a hit die of 8. The formula should look like this: hp = level * (conModifier + hitDie).
  4. The character's AC (armor class) uses a different formula depending on their class:
    • A fighter's AC is equal to 18
    • A ranger's AC is equal to 15 plus their dexterity modifier, but their AC cannot be higher than 17
    • A rogue's AC is equal to 12 plus their dexterity modifier
  5. The character's proficiency is not displayed, but it is used to calculate their attack bonus. It uses the formula proficiency = 1 + (level + 3) / 4.
  6. Depending on their class, the character has one or two attacks. Each attack has a name and a bonus. The bonus is equal to the character's proficiency bonus plus one of their ability modifiers.
    • A fighter has one attack called Greatsword, which uses their strength modifier
    • A rogue has two attacks: Dagger and Crossbow, which both use their dexterity modifier
    • A ranger has two attacks: Longsword, which uses their strength modifier, and Longbow, which uses their dexterity modifier
