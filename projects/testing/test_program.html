The Program

This section explains what the program is and includes some details that you don't need in order to use it. This information is here in case you're curious about how the program works, and it may help you troubleshoot any problems that come up when using it.

Summary

The test program is written in Java and launched by a shell script called test.sh. This script runs tests/bin/projecttests/Main and passes along any arguments you provide (everything you type after sh tests/test.sh).

I've only tested this program on bash terminals. It should work fine on bash, zsh, or a bash emulator running on Windows. You might be able to run it from cmd or powershell by using the same command found in test.sh, but I don't know whether every part of the program will work correctly.

Tests Directory

Below is a view of the projects directory, the tests directory, and their contents (assuming you set up the project folders and unpacked the test program correctly):

projects/
├── final/
├── project1/
├── project2/
├── project3/
└── tests/
    ├── bin/
    ├── logs/
    ├── res/
    ├── src/
    ├── student/
    ├── build
    ├── LICENSE.txt
    ├── README.txt
    └── test.sh

The logs and student directories are not included in the zip, but the program will create them when it runs.

You should never need to edit anything in the tests directory or any of its subdirectories. The test program may overwrite any files inside of the tests directory or its subdirectories, so don't keep any of your own files in there. As long as you know not to use the tests directory for your own documents, you don't need to be familiar with anything inside that folder to use the program.

  • bin contains the compiled Java code (binary files such as .class and .jar files) for the test programs
  • res contains resources used by the tests, such as inputs, expected outputs, and supplementary jshell scripts
  • logs contains logs generated by the test program
  • src contains the Java source code for the tests
  • student contains modified copies and compiled versions of your code used by the tests, backup copies so the status command can check for changes, and a record of the last test for each project
  • build is the bash script that I use to compile the test program
  • LICENSE.txt is the text of the MIT license, which applies to this test program
  • README.txt is a text file containing some of the documentation provided here
  • test.sh is the aforementioned shell script that launches the test program

Test Logs

When the test program runs, it will write information about what it's doing to a log file. This file isn't usually going to be helpful unless I need to help you troubleshoot an unusual problem or find and fix a bug in my test code. All of these files are saved to the tests/logs folder.

Backup Copy of Student Code

As mentioned above, the program saves a backup copy of your code so that the status command can check whether your program has changed since it was last tested. While I'm calling this a "backup copy," you should not rely on it to back up your work for a few reasons:

  • Only the most recently-tested version of your project is saved, and the backup file(s) are overwritten every time you run the test command with that project.
  • The tests folder is in the same location as your project folders, which means a mishap that causes you to lose your projects will probably delete the tests folder and any backup copies it contains.
  • If you start editing this file, then your work will be lost (probably with no way to recover it) as soon as you run the test command with that project. This is why I cautioned you earlier not to keep any of your own files inside of the tests folder.

Keep your work backed up separately with a cloud storage service such as Dropbox, Google Drive, or OneDrive. If you know how to use version control software such as git, then keep your work backed up in a remote repository. If you lose work that was saved when testing your program, then it won't hurt to check whether the backup copy includes some of the lost work, but don't count on it helping you prevent data loss. That's not why it's there, and I don't want it to give anyone a false sense of security.
